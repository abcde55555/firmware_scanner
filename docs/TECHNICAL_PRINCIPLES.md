# RTOS Firmware Analyzer 技术原理文档

## 1. 项目概述

**rtos-firmware-analyzer** 是一个专业的 RTOS 固件安全分析工具，能够对嵌入式/物联网设备固件进行自动化解包、架构识别、组件提取和 SBOM 生成。

### 适用场景
- 固件安全漏洞评估（识别有漏洞的第三方组件版本）
- 供应链安全审计（生成完整的软件物料清单）
- 嵌入式产品合规审查（组件许可证识别）
- 竞品分析（了解竞品使用了哪些开源组件）

### 适用目标
- 物联网设备固件（传感器、网关、智能家居）
- 工控设备固件（PLC、SCADA）
- 消费电子固件（路由器、摄像头、手环）
- 汽车电子固件（ECU、T-Box）

## 3. 固件格式结构详解

### 3.1 ELF 格式

ELF（Executable and Linkable Format）是嵌入式Linux和RTOS最常用的可执行文件格式。

```
┌─────────────────────┐  偏移 0x00
│   ELF Header        │  ← 52/64 字节，包含 e_machine (架构)
│   e_ident: 7f 45 4c 46 │
│   e_machine: ARM=0x28   │
│   e_entry: 入口地址     │
├─────────────────────┤
│ Program Headers     │  ← 描述段加载信息
├─────────────────────┤
│ .text (代码段)      │  ← 可执行指令
├─────────────────────┤
│ .rodata (只读数据)  │  ← 字符串常量、版本信息
├─────────────────────┤
│ .data (已初始化数据)│
├─────────────────────┤
│ .bss (未初始化数据) │
├─────────────────────┤
│ .symtab (符号表)    │  ← 函数名、变量名
├─────────────────────┤
│ .strtab (字符串表)  │
├─────────────────────┤
│ Section Headers     │
└─────────────────────┘
```

**识别方式**：
- 解析 `e_machine` 字段确定 CPU 架构
- 解析 `.symtab` 获取所有函数名，匹配已知 RTOS API
- 扫描 `.rodata` 段中的版本字符串

### 3.2 ESP-IDF 镜像格式

```
┌─────────────────────┐  偏移 0x00
│ Image Header (24B)  │
│   magic: 0xE9       │
│   segment_count     │
│   entry_point       │
├─────────────────────┤
│ Segment 0 Header(8B)│  load_addr + size
│ Segment 0 Data      │  代码/数据
├─────────────────────┤
│ Segment 1 Header    │
│ Segment 1 Data      │
├─────────────────────┤
│ ...                 │
├─────────────────────┤
│ Checksum / Hash     │
└─────────────────────┘
```

**识别方式**：
- 首字节为 `0xE9`
- segment_count 在 1-16 范围内
- entry_point 指向 0x4000xxxx（IRAM 区域）

### 3.3 U-Boot 镜像格式

```
┌─────────────────────┐  偏移 0x00
│ Header (64 bytes)   │
│   magic: 27 05 19 56│
│   data_size         │
│   load_addr         │
│   entry_point       │
│   os_type           │
│   arch_type         │
│   compress_type     │
│   name[32]          │
├─────────────────────┤
│ Payload (可能压缩)  │
│   gzip / lzma / raw │
└─────────────────────┘
```

### 3.4 原始二进制（Raw Binary）

没有文件头的纯机器码文件。通过以下方式分析：

1. **熵分析**：计算每4KB块的香农熵
   - 熵 < 1.0：填充区域（0x00/0xFF）
   - 熵 5.0-7.0：代码区域
   - 熵 > 7.0：压缩/加密数据

2. **向量表检测**（ARM Cortex-M）：
   - 偏移 0x00：初始栈指针（0x2000xxxx 范围 = RAM）
   - 偏移 0x04：复位处理函数（0x0800xxxx = Flash，且为奇数 = Thumb模式）

## 4. RTOS 固件结构特征

### 4.1 FreeRTOS

```
典型内存布局：
┌─────────────────┐ 0x08000000 (Flash)
│ 向量表          │ ISR 向量
├─────────────────┤
│ 启动代码        │ SystemInit, main()
├─────────────────┤
│ FreeRTOS 内核   │ 调度器、队列、信号量
├─────────────────┤
│ 应用任务代码    │ 用户任务
├─────────────────┤
│ 库代码          │ lwIP, mbedTLS 等
├─────────────────┤
│ .rodata         │ 字符串、常量
└─────────────────┘
┌─────────────────┐ 0x20000000 (RAM)
│ TCB 数组        │ 任务控制块
├─────────────────┤
│ 任务栈空间      │ 每个任务独立栈
├─────────────────┤
│ 堆内存          │ pvPortMalloc 分配
└─────────────────┘
```

**特征识别**：
| 特征类型 | 具体内容 | 置信度 |
|---------|---------|--------|
| 字符串 | `"FreeRTOS"`, `"Tmr Svc"`, `"IDLE"` | 高 |
| 函数名 | `xTaskCreate`, `vTaskStartScheduler`, `pvPortMalloc` | 高 |
| 版本字符串 | `"FreeRTOS V10.5.1"` | 高 |
| 内部变量 | `pxCurrentTCB`, `xTickCount` | 高 (Ghidra) |

### 4.2 Zephyr RTOS

**特征识别**：
- 设备树标识：`__device_dts_ord_` 开头的符号
- 内核API：`k_thread_create`, `k_sem_init`, `z_swap`
- 构建系统标识：`CONFIG_KERNEL`, `ZEPHYR_BASE`
- 版本字符串：`"Zephyr v3.x.x"`

### 4.3 RT-Thread

**特征识别**：
- 函数前缀：`rt_thread_create`, `rt_sem_create`, `rt_device_register`
- Shell标识：`"finsh"`, `"msh>"`
- 内核对象 magic：`0x23232323`

### 4.4 VxWorks

**特征识别**：
- 厂商标识：`"Wind River"`, `"VxWorks"`
- 系统API：`taskSpawn`, `semBCreate`, `msgQCreate`
- 调试代理：`"WDB_AGENT"`

### 4.5 ThreadX (Azure RTOS)

**特征识别**：
- 厂商标识：`"Azure RTOS"`, `"ThreadX"`
- 内核API：`tx_thread_create`, `tx_kernel_enter`
- 内部变量：`_tx_thread_current_ptr`

## 5. 识别技术原理

### 5.1 字符串模式匹配

**原理**：遍历固件二进制中的所有可打印 ASCII 字符串（最小长度4），对每个字符串执行正则表达式匹配。

```
固件数据: ... 00 46 72 65 65 52 54 4F 53 20 56 31 30 2E 35 2E 31 00 ...
                F  r  e  e  R  T  O  S     V  1  0  .  5  .  1
                            ↓
           正则: FreeRTOS\s+[Vv](\d+\.\d+\.\d+)
                            ↓
           结果: FreeRTOS, 版本 10.5.1
```

**覆盖范围**：200+ 组件的版本字符串模式
**优势**：速度快，无需反汇编
**局限**：需要固件中存在明文版本字符串

### 5.2 符号表解析

**原理**：解析 ELF 文件的 `.symtab`（符号表）和 `.dynsym`（动态符号表），获取所有函数名和全局变量名，与已知组件的 API 函数名数据库做匹配。

```
ELF 符号表:
  [1234] xTaskCreate        FUNC  GLOBAL  .text
  [1235] vTaskDelay         FUNC  GLOBAL  .text
  [1236] pvPortMalloc       FUNC  GLOBAL  .text
          ↓
  匹配 FreeRTOS 已知符号库 → 命中 3/15 → 置信度 0.6
```

**优势**：准确度高，即使没有版本字符串也能识别组件存在
**局限**：仅适用于未 strip 的 ELF 文件

### 5.3 二进制签名匹配

**原理**：在原始字节数据中搜索已知的字节序列特征。与字符串匹配不同，这里处理的是字节级别的 pattern（包含不可打印字符）。

**关键实现细节**：
- 短模式（<8字节）需要词边界检查，避免误匹配
- 多次命中同一签名会提升置信度
- 记录命中偏移作为证据链

### 5.4 反汇编识别（Capstone）

**原理**：使用 Capstone 反汇编引擎将二进制代码反汇编为汇编指令，识别特定的指令序列模式。

```
ARM Cortex-M 上下文切换识别：
  MRS    R0, PSP         ; 读取进程栈指针
  STMDB  R0!, {R4-R11}   ; 保存寄存器
  STR    R0, [R1]        ; 存入 TCB
  LDR    R0, [R2]        ; 加载新任务 TCB
  LDMIA  R0!, {R4-R11}   ; 恢复寄存器
  MSR    PSP, R0         ; 切换栈指针
  
  → 识别为 RTOS 上下文切换代码
```

**优势**：可识别无符号、无字符串的固件中的 RTOS 代码
**局限**：需要先确定架构，计算量较大

### 5.5 邻近版本检测

**原理**：当通过签名匹配确认了某组件的存在，在该命中点前后 512 字节范围内搜索版本号模式。

```
偏移 0x1A00: "mbedtls_ssl_init"  ← 组件签名命中
                 ↓ 搜索范围 [0x1800, 0x1C00]
偏移 0x1A40: "3.4.0"             ← 找到版本号
                 ↓
结果: mbedTLS 3.4.0, 置信度 0.85
```

**回退策略**：
1. 邻近搜索 → 2. 全局搜索 `"ComponentName vX.Y.Z"` → 3. 搜索 `#define VERSION` → 4. 拼接 MAJOR.MINOR.PATCH

### 5.6 深度反编译（Ghidra / radare2）

**原理**：使用专业逆向工具进行完整的程序分析：
- 反编译为 C 代码，在伪代码中搜索版本字符串
- 识别全局变量名（如 `pxCurrentTCB`）
- 分析函数调用图确定组件边界
- 导入表分析

**优势**：最高准确度，可发现其他方法无法找到的组件
**局限**：需要安装外部工具，分析速度慢

## 6. 完整分析流程

```
输入: firmware.bin
  │
  ▼ Step 1: 加载
  读取文件 → 计算 SHA-256/MD5 → 验证大小
  │
  ▼ Step 2: 格式检测
  所有 Format Handler 的 can_handle() 打分
  选择最高分的 handler → unpack() 解包为段
  │
  ▼ Step 3: 架构检测（多策略投票）
  ┌─ ELF头解析 (置信度 0.95) ─ 直接读取 e_machine
  ├─ Magic bytes (置信度 0.75) ─ 向量表、固定模式
  └─ Capstone探测 (置信度 0.70) ─ 试解码，看成功率
  → 选最高置信度结果
  │
  ▼ Step 4: RTOS 检测
  9个 RTOS 插件各自打分 → 选最高分
  │
  ▼ Step 5: 组件提取（多引擎并行）
  ┌─ StringPatternExtractor (优先级80)
  ├─ SymbolTableExtractor   (优先级70)
  ├─ BinarySignatureExtractor (优先级60)
  ├─ DisassemblyExtractor   (优先级40)
  ├─ Radare2Extractor       (优先级30, 可选)
  └─ GhidraExtractor        (优先级20, 可选)
  │
  ▼ Step 6: 深度扫描（多线程，每段独立）
  ThreadPoolExecutor 并行扫描每个 section
  200+ 签名 × 每个段 → 命中列表
  邻近版本检测 → 版本解析
  │
  ▼ Step 7: 交叉验证
  相同组件的多个检测结果 → 加权投票
  权重: Ghidra(0.95) > r2(0.90) > 符号表(0.85) > 反汇编(0.75) > 签名(0.80) > 字符串(0.60)
  多源一致 → 置信度 ×1.2 提升
  │
  ▼ Step 8: 输出
  → CycloneDX SBOM (JSON)
  → HTML 可视化报告
  → JSON 原始数据
```

## 7. 扫描方式说明

### 不是文件系统遍历

RTOS 固件通常**没有文件系统**，不像 Linux 那样有 `/lib/`, `/usr/bin/` 等目录结构。RTOS 固件是一个**单一的二进制 blob**，所有代码和数据链接为一个整体。

### 本工具的扫描方式

1. **二进制级别扫描**：直接在字节流中搜索特征模式
2. **结构化解析**：如果是 ELF，利用其节区结构定位代码/数据/符号
3. **段独立分析**：解包后的每个段（section）独立扫描
4. **熵引导**：用熵分析区分代码区、数据区、压缩区
5. **递归分析**：发现压缩区域自动解压后再扫描
6. **限制保护**：每段最大扫描 4MB，超时 30 秒自动跳过

### 与传统固件分析工具的区别

| 特性 | binwalk | 本工具 |
|------|---------|--------|
| 目标 | 提取文件系统 | 识别组件+版本 |
| 输出 | 文件/目录 | SBOM + 报告 |
| 深度 | 格式识别 | 组件+版本+置信度 |
| RTOS支持 | 有限 | 专门优化 |
| 多方法交叉验证 | 无 | 6种方法加权投票 |

## 8. 置信度与交叉验证机制

### 方法权重

```
Ghidra反编译        0.95  ← 最高，代码级别分析
radare2分析         0.90
符号表匹配          0.85  ← ELF 专用，准确度高
二进制签名          0.80
反汇编模式          0.75
字符串模式          0.60  ← 最低，可能有误匹配
```

### 置信度计算

```python
# 单一方法的置信度
confidence = base_confidence × method_weight

# 多方法检测同一组件 → 叠加
total_confidence = sum(每次检测的 weighted_confidence)

# 多源增强
if 2种以上方法确认同一版本:
    total_confidence × 1.2
if 3种以上方法确认:
    total_confidence × 1.1 (额外)
```

### 版本冲突解决

当不同方法检测到同一组件的不同版本：
1. 对每个版本号按方法权重求和
2. 选择总权重最高的版本
3. 保留所有检测证据链供人工审核
